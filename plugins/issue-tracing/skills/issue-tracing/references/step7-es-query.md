# Step 7 — Run ES queries (per-project counts)

Required filters on every query:
- `range` on `@timestamp` — the parsed time range with ±5 min buffer
- an `env` filter — usually `prod` (from URL filters)
- a `project` filter — REQUIRED; one query per candidate project
- a `level` filter — default `["error", "fatal"]`; include `"warning"` only if the originating alert was warning-level or the dashboard panel includes warnings. Count warning separately from error/fatal.

**Copy the URL's filter clause verbatim — do NOT default to `term .keyword`.** Kibana URL filters carry the exact query, e.g. `match_phrase:(project:<p>)` OR `match_phrase:(project.keyword:<p>)` — the field path differs per data view, the type is always `match_phrase`. Replicate it as-is; do not "upgrade" to `term <field>.keyword`. Why: many data streams map `project` / `env` / `level` as plain `text` with **no `.keyword` sub-field**, and a `term <field>.keyword` on a missing sub-field matches nothing → a false `0` with no error (this has burned whole investigations). `match_phrase` on whichever field the URL names sidesteps this, and also avoids value-casing misses like `"Error"` vs `"error"`. Only build your own `term <field>.keyword` after confirming the sub-field exists (sample one doc with `_source: "*"` or `get_mappings`). For a multi-value filter, use a `bool.should` of `match_phrase` clauses (mirroring the URL's `bool.should`), not `terms`.

**NEVER run a query without a project filter, time range, AND level filter.** A `*` or `match_all` query is forbidden — past incidents include heap exhaustion from unbounded queries.

**DO NOT include `aggs` / `aggregations` in your search body.** The `mcp__elasticsearch__search` wrapper strips aggregation results from responses; only `hits` are returned. Writing an `aggs` block wastes the round-trip — the wrapper accepts the body but drops the results, so you can't see what you asked for. Use per-bucket queries instead:
- Total counts: `size: 0` **with `track_total_hits: true`** — then read the `Total results: N` line. WITHOUT `track_total_hits: true` the total is capped at 10000, so a real 72k count silently reads as "10000" and every baseline ratio / impact number built on it is wrong. Never use a `count` / `value_count` agg (stripped by the wrapper).
- Top message patterns: `size: 5` sorted by `@timestamp desc`, **read the message out of `_source`** — the message text comes back in every hit's `_source`, so you already have it from your sample. Do NOT try to *filter* on the message field to count patterns (see the dominant-pattern note below).
- First / last occurrence: two queries with `sort` `asc` and `desc`, `size: 1`
- Distinct user counts: see the distinct-user handling in the log-analysis step (cannot use `cardinality` agg)

**Token-budget rules** (the wrapper truncates large responses to a file when responses exceed its limit, costing extra `jq` round-trips):
- **Always start with `size: 0`** to get the total. Only sample after that.
- **`size` cap when fetching `message`/stack traces: 5.** A single `size: 100` over `error` logs with stack traces typically blows the limit. If 5 is not enough, paginate (`from`+`size`) or refine the filter — do not raise `size`.
- **Always pass `_source`** with only the fields you need. For project distribution use `_source: ["project"]`; for time check use `_source: ["@timestamp"]`. Default `_source: "*"` only when you need to inspect schema once.
- **Dedupe stack traces.** When sampling errors, identify each unique error pattern by its leading message (first line / exception type). Once you have one full sample per pattern, do NOT pull additional documents that share the same pattern — re-reading the same stack trace 5 times costs tokens for zero new information. Use `must_not match_phrase` to exclude already-seen patterns when fetching the next sample — **but only if the message field is actually searchable** (see the dominant-pattern note; on server-log streams where it is not, this clause silently no-ops, so dedupe client-side from `_source` instead).

**Finding the dominant pattern (do NOT trust the first sample)**:
- **DEFAULT FIRST MOVE — bucket client-side in ONE pass, do NOT loop server-side message counts.** Pull one bounded set (`_source: ["message", …]`; let the wrapper dump to a file if it is large) and bucket the leading lines with `grep` / `jq` locally. That is a single round-trip and it gives you the whole distribution at once. Firing N separate `size: 0` `match_phrase` message counts (one per candidate pattern) is the wrong reflex: each is a round-trip that eats the ≤2–3 concurrency ceiling, AND the message field tokenizes camelCase / dotted identifiers unreliably (`QuantumApi`, `InJoiEntryProxy`, `RedeemAuthorizationCodeAsync` can all silently return `0` even though they are right there in `_source`). Reserve server-side counts for *confirming one specific share after* client-side bucketing — never as the primary weighting loop.
- The first message you sample is NOT necessarily the dominant pattern — recent errors are biased toward whatever fired most recently, not most often.
- **Weight patterns from `_source`; do NOT filter on the message field.** In filebeat-style server-log data streams the `message` / `body` field is typically stored-but-not-analyzed (and has no usable `.keyword`), so `match` / `match_phrase` / `wildcard` on it silently return **`0` even though the text is right there in `_source`** — a false negative that can make a 100-hit pattern read as "0 weight". You do not need to query it anyway: the message came back in `_source` on the hits you already sampled. To weight / classify, pull the bounded hit set with `_source: ["message", …]` and bucket the leading lines **client-side** (`grep` / `jq`); if the set is large enough to blow the token budget, let the wrapper dump to a file and `grep` / `jq` that. Classify by a structured field (endpoint / logger / exception type) when one exists.
- Only once you have **confirmed the message field IS searchable** — a `match_phrase` on a substring you just saw in `_source` returns non-zero — may you weight patterns with `size: 0` `match_phrase` counts. **Never conclude a pattern's share from a `message` query you have not proven returns non-zero on a known-present string.**
- If a candidate covers a meaningful share (e.g. ≥10% of total), treat it as a main pattern; if <10%, drop it (client-side, or `must_not match_phrase` when searchable) and look again.
- This avoids two failure modes: "first sample was 2/2151, agent built whole report around it" AND "`match_phrase` on message returned 0, so I assumed the pattern didn't exist".
